# Plan 27-03: DynamoDB Schema Documentation + Migration Strategy

<objective>
Document all DynamoDB tables with keys, GSIs, attributes, and access patterns. Create migration strategy for schema changes with version tracking.
</objective>

<tasks>

<task id="1" name="Document DynamoDB schema">
<read_first>
- secure-db.js
- workspace-metadata-store.cjs
- src/config/index.js
</read_first>
<action>
Create `docs/dynamodb-schema.md` documenting:

1. **Users Table:**
   - Table name: from config `MESH_DYNAMO_TABLE_USERS`
   - Partition key: `pk` (string — user ID)
   - Sort key: `sk` (string — record type)
   - Attributes: email, passwordHash, createdAt, updatedAt
   - Access patterns: get by ID, get by email (GSI needed?)
   - Encryption: AES-256-GCM for sensitive fields (via secure-db.js)

2. **Sessions Table:**
   - Table name: from config
   - Partition key: session ID
   - TTL attribute for automatic expiration
   - Access patterns: get/create/delete by session ID

3. **User Store Table:**
   - Table name: from config
   - Partition key: user ID
   - Sort key: store key
   - Access patterns: get/put user preferences, credentials (encrypted)

4. **Workspace Metadata Table:**
   - Table name: from config `MESH_DYNAMO_TABLE_METADATA`
   - Partition key: workspace ID
   - Sort key: file path
   - Attributes: hash, size, lastModified, mimeType
   - GSI: by workspace for listing
   - Access patterns: get file metadata, list workspace files, batch update

For each table: list all attributes with types, all GSIs with key schemas, all access patterns with the function that performs them.

<task id="2" name="Create migration strategy document">
<read_first>
- secure-db.js
- infra/cloudformation.yml
</read_first>
<action>
Create `docs/migration-strategy.md` documenting:

1. **Adding new attributes:**
   - DynamoDB is schema-less — just start writing the new attribute
   - Update `secure-db.js` or `workspace-metadata-store.cjs` to read/write new field
   - Default handling for records without the new attribute (backward compatibility)

2. **Adding new tables:**
   - Add to CloudFormation template
   - Add table name to `src/config/index.js` env vars
   - Create access functions in secure-db.js or new data access module

3. **Modifying existing attributes:**
   - Rename: write both old+new during transition, then drop old
   - Type change: migration script to batch-update existing records
   - Index changes: create new GSI, verify, delete old GSI

4. **Schema version tracking:**
   - Add `SchemaVersion` item in a config/metadata table
   - Migration scripts check version and apply changes sequentially
   - Pattern:
     ```javascript
     const CURRENT_SCHEMA_VERSION = 1;
     async function migrateIfNeeded(db) {
       const version = await db.getSchemaVersion();
       if (version < CURRENT_SCHEMA_VERSION) {
         // Apply migrations sequentially
       }
     }
     ```

5. **Safety rules:**
   - Never delete attributes or tables without a migration window
   - Always add backward-compatible defaults for new attributes
   - Test migrations against DynamoDB Local before production
   - CloudFormation handles table creation; migration scripts handle data
</action>
