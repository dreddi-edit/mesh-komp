# Plan 28-02: Fix Monaco Editor Interactivity and CSP Blob Workers

## Objective

Ensure the Monaco editor initializes as a fully interactive code editor with syntax highlighting, IntelliSense, editing, undo/redo. The likely root cause is CSP blocking blob: URLs needed for Monaco web workers, causing silent fallback to degraded mode. Also ensure files load completely without truncation.

## must_haves

1. Monaco editor is fully interactive (not read-only or degraded)
2. Monaco web workers load successfully (no CSP violations in console)
3. Files open with complete content (no truncation)
4. Syntax highlighting works across all supported languages

## Tasks

<task id="1">
<title>Audit and fix CSP directives for Monaco blob workers</title>
<read_first>
- src/server.js (find helmet/CSP configuration — look for contentSecurityPolicy, script-src, worker-src directives)
- views/app.njk (lines 482-495 — MonacoEnvironment getWorkerUrl configuration)
</read_first>
<action>
In `src/server.js`, find the Content-Security-Policy configuration (helmet or manual headers). Ensure the following directives include `blob:`:

- `worker-src`: Must include `'self'` and `blob:` — Monaco creates blob URLs for editor workers (language, editor, typescript, json, css, html workers)
- `child-src`: Must include `'self'` and `blob:` — fallback for older browsers
- `script-src`: Must include the Monaco CDN origin `https://cdn.jsdelivr.net` — Monaco loads scripts from this CDN

The exact CSP directives needed for Monaco 0.52.2 via CDN:

```javascript
// In helmet or manual CSP header configuration:
'worker-src': ["'self'", 'blob:'],
'child-src': ["'self'", 'blob:'],
'script-src': ["'self'", "'unsafe-eval'", 'https://cdn.jsdelivr.net', 'blob:'],
```

Note: Monaco requires `unsafe-eval` for its parser/tokenizer. This is a known Monaco requirement. If the project uses nonces, the nonce must also be on the Monaco CDN script tags in `views/app.njk`.

Search for where CSP headers are set — could be in:
- `helmet()` middleware configuration
- Manual `res.setHeader('Content-Security-Policy', ...)` calls
- A middleware function that sets security headers
</action>
<acceptance_criteria>
- CSP header includes `blob:` in `worker-src` directive (verify with: `grep -n "blob:" src/server.js`)
- CSP header includes `https://cdn.jsdelivr.net` in `script-src` directive
- Browser console shows zero CSP violation errors when loading the app page
- Monaco editor shows syntax highlighting (colored tokens, not plain white text)
</acceptance_criteria>
</task>

<task id="3">
<title>Verify file content loading is complete (not truncated)</title>
<read_first>
- assets/app-workspace.js (lines 1197-1200 — openFile function)
- assets/app-workspace.js (lines 1174-1185 — switchTab function)
</read_first>
<action>
The `openFile()` function at line 1199 uses `await item.handle.getFile()` then `await f.text()` which reads the full file via the File System Access API. This should NOT truncate content.

However, the tab model creation at line 1178 uses `t.content` which was stored at file open time. If `t.content` is somehow truncated (e.g., the File object was garbage collected before `.text()` resolved), the Monaco model gets truncated content.

Fix: Create the Monaco model immediately with the file content rather than deferring:

```javascript
async function openFile(item){
  if(S.tabs.find(x=>x.path===item.path)){switchTab(item.path);return;}
  try{
    const f=await item.handle.getFile();
    const txt=await f.text();
    const lang = langOf(item.path);
    const model = S.monacoReady ? monaco.editor.createModel(txt, lang) : null;
    S.tabs.push({path:item.path, content:txt, model, handle:item.handle});
    switchTab(item.path);
  }catch(e){toast('Error','Cannot read '+item.name);}
}
```

And in `switchTab`, skip model creation if it already exists:

```javascript
// In switchTab (line 1178):
if(S.editor){
  if(!t.model) t.model = monaco.editor.createModel(t.content||'', langOf(path));
  S.editor.setModel(t.model);
}
```

This is already the existing behavior — the model is created lazily. The fix above just eagerly creates it when Monaco is ready.

Also add a diagnostic log to verify content size matches file size:

```javascript
// In openFile, after reading:
if(txt.length < f.size * 0.9) console.warn('[mesh] File content may be truncated:', item.path, 'read:', txt.length, 'expected:', f.size);
```
</action>
<acceptance_criteria>
- `openFile` function creates Monaco model eagerly when `S.monacoReady` is true
- Large files (>100KB) open with complete content in the editor
- No `[mesh] File content may be truncated` warnings in console
</acceptance_criteria>
</task>
