# Plan 28-05: Fix Back-Navigation in Settings

## Objective

All settings views have `href="app"` (relative path, no leading slash) on the topbar logo and "← Workspace" back link. When the user is on a standalone page like `/settings-account`, this resolves to `/settings-account/app` (a 404 path) before the JS override in `settings.js` runs. Fix: change all occurrences to `href="/app"` (absolute). The JS-based `applyStandaloneNavigation()` override still applies the `?returnTo` value correctly at runtime — this just fixes the fallback path.

Also fix inter-settings nav links in standalone pages that point to individual pages (e.g., `href="/settings-appearance"`) — these bypass the combined SPA. Update them to point to `/settings#{section}` so clicking a nav item lands on the correct hash in the combined SPA.

## must_haves

1. Clicking "← Workspace" from any settings page navigates to `/app` (not a 404 or login redirect)
2. All 7 settings views have `href="/app"` on topbar-back and topbar-logo links
3. Inter-settings nav links in standalone pages resolve to `/settings#{section}` not `/settings-{page}`
4. The combined `settings.njk` SPA still works correctly via JS hash routing
