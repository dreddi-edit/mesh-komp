# Plan 08-01: Fix Compression Analytics Data Pipeline

## Objective

Ensure `S.compressionMap` is populated with real server-side compression data in all
folder-open scenarios, including page reload with an already-indexed workspace (where the
sync skip-gate means `compressionStats` is empty from sync batches).

## Background

The research identified three gaps in the current implementation:

1. `mesh-indexing-initial-ready` fires when all files match (skip gate) but does NOT
   trigger `loadCompressionMap()`. Only `mesh-indexing-complete` does.
2. After `openFolder` / `restoreFolder` completes the background index cycle, `renderTree()`
   is called with seed (placeholder) data — `loadCompressionMap()` is not called there.
3. The `loadCompressionMap()` at bootstrap (line 1938) runs before `restoreFolder()` which
   means on the first render `S.dirName` may still be empty — the tree re-render covers this,
   but only if the user had a prior session.

<task id="08-01-01">
  <title>Add loadCompressionMap on mesh-indexing-initial-ready event</title>
  <wave>1</wave>

  <read_first>
    - assets/app-workspace.js — full file, specifically lines 229-251 (skip-gate path that fires mesh-indexing-initial-ready) and lines 1953-1958 (existing mesh-indexing-complete handler)
  </read_first>

  <action>
  Add a new event listener for `mesh-indexing-initial-ready` immediately after the existing
  `mesh-indexing-complete` listener (around line 1958 in app-workspace.js).

  Insert this code block after the closing `});` of the `mesh-indexing-complete` handler:

  ```js
  window.addEventListener('mesh-indexing-initial-ready', () => {
    loadCompressionMap().then(() => {
      renderTree();
      if (S.currentView === 'ops') renderOps();
    });
  });
  ```

  This covers the skip-gate path: when all files have identical SHA-256 digests, the sync
  returns early and fires `mesh-indexing-initial-ready` (line 235-239 in the skip path).
  Without this handler, `S.compressionMap` stays seeded with placeholder `{capsuleBytes:0}`
  entries from `seedCompressionMapFromTree()` and tooltips always show 0%.
  </action>

  <acceptance_criteria>
    - `assets/app-workspace.js` contains `window.addEventListener('mesh-indexing-initial-ready'`
    - The handler calls `loadCompressionMap().then(()` and `renderTree()`
    - The new listener appears after the `mesh-indexing-complete` listener
  </acceptance_criteria>
</task>

  <action>
  In the `openFolder` function, inside the `deepScanAll(S.tree).then(async()=>{ ... })` callback,
  add a call to `loadCompressionMap()` after the background `runWorkspaceIndexCycle` completes.

  Current code at end of the deepScanAll callback (around line 644-646):
  ```js
      await runWorkspaceIndexCycle('background', { scanEpoch: S.workspaceIndex.scanEpoch, complete: true, deferReadyState: true });
      await initMeshMetadata(h, { force: true, phase: 'background-complete', attachToTree: true });
      updateIndexProgressState('graph-ready', { ratio: 1 });
  ```

  Change to:
  ```js
      await runWorkspaceIndexCycle('background', { scanEpoch: S.workspaceIndex.scanEpoch, complete: true, deferReadyState: true });
      await initMeshMetadata(h, { force: true, phase: 'background-complete', attachToTree: true });
      updateIndexProgressState('graph-ready', { ratio: 1 });
      loadCompressionMap().then(() => { renderTree(); if (S.currentView === 'ops') renderOps(); });
  ```

  Note: `mesh-indexing-complete` fires inside `runWorkspaceIndexCycle` which will also trigger
  the event-based handler from task 08-01-01. But for the case where the background cycle fires
  `mesh-indexing-complete` BEFORE `deferReadyState` resolves, having the explicit call here
  ensures the tree re-render happens with real data even if the event fires before this point.
  The `loadCompressionMap` function itself is idempotent — calling it twice is fine because
  it only overwrites entries where `incomingRaw > existing.rawBytes`.
  </action>

## Verification

Manual browser verification (no automated test possible — requires live browser + workspace):

1. Open a folder that has been indexed before (existing workspace):
   - Wait for "Graph ready" state
   - Open browser console → `S.compressionMap.size` should be > 0
   - `[...S.compressionMap.values()].some(e => e.capsuleBytes > 0)` should be true

2. Hover a file in the explorer sidebar → tooltip should show e.g. `42% compressed (1.2KB → 0.7KB)`

3. Hover a folder → tooltip should show e.g. `38% avg compression (14 files)`

4. Switch to ops view → table should show non-zero "Compressed" and "Ratio" columns

5. Hard-reload the page with the same workspace already indexed:
   - Compression data should appear WITHOUT needing to trigger a re-index
