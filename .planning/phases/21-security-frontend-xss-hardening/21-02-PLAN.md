# Plan 21-02: High-Risk DOM Injection Remediation

<objective>
Replace all HIGH RISK unsafe DOM content injection in the three largest frontend files with safe DOM APIs (createElement+textContent) or DOMPurify.sanitize(). Targets: chat messages, file trees, graph labels, user-content display points.
</objective>

<tasks>

<task id="1" name="Harden app-workspace.js">
<read_first>
- assets/app-workspace.js
- assets/dom-utils.js
</read_first>
<action>
In `assets/app-workspace.js` (1,957 lines):

1. Include dom-utils.js script in app.html if not already loaded
2. Find all raw content injection assignments that render user content:
   - Chat messages (approx line 1216 area) — ensure ALL chat message rendering goes through sanitizeHtml()
   - File tree rendering — file names from API could contain special chars, use safeEl() for text nodes
   - Workspace name display — user-provided names, use textContent assignment
   - Error messages displayed to user — use textContent assignment
3. For each HIGH RISK site: replace raw content assignment with safeHtml(el, content) or el.textContent = content (prefer textContent when no HTML needed)
4. For markdown-rendered content (already HTML from marked), wrap with sanitizeHtml() to strip unsafe tags
5. Do NOT change static template construction (LOW RISK sites)
</action>
<acceptance_criteria>
- User-content injection sites in app-workspace.js use safeHtml/safeEl/textContent
- Chat messages render correctly with HTML formatting preserved
- File names with special characters display as text, not executed
</acceptance_criteria>
</task>

<task id="2" name="Harden app.js">
<read_first>
- assets/app.js
- assets/dom-utils.js
</read_first>
<action>
In `assets/app.js` (871 lines):

1. Find all raw content injection with user-derived content
2. Replace user-content injection with safeEl() or safeHtml() from dom-utils.js
3. Focus on: notification messages, status displays, dynamic UI labels
4. Keep static template HTML construction that uses only hardcoded strings
</action>
<acceptance_criteria>
- User-content injection sites in app.js use safe DOM APIs
</acceptance_criteria>
</task>

<task id="3" name="Harden app-graph.js">
<read_first>
- assets/app-graph.js
- assets/dom-utils.js
</read_first>
<action>
In `assets/app-graph.js` (851 lines):

1. Graph node labels may display file names from API — use textContent for text labels
2. Tooltip/popover content with file paths — use safeEl() or textContent
3. D3 integration: if using .html() for node labels, switch to .text() for user data
4. SVG text elements are naturally XSS-safe but verify no raw injection on SVG foreignObject elements
</action>
<acceptance_criteria>
- Graph labels with file names use textContent or D3 .text()
- No raw content injection with user-derived file paths in app-graph.js
</acceptance_criteria>
</task>

</tasks>

<verification>
- Chat messages, file trees, and graph render correctly
- Injecting malicious filenames displays as text, not executed
- No visual regressions in app, workspace, or graph views
</verification>

<must_haves>
- All user-content DOM injection replaced with safe DOM APIs or DOMPurify
- Chat, file tree, graph label rendering is XSS-safe
</must_haves>
