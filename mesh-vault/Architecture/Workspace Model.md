# Workspace Model

## Two Workspace Source Kinds

### `local-path`
- User selects a local folder in the browser (File System Access API)
- Worker opens and operates on the real path on disk
- Terminal opens directly in the real root path
- Git operations run against `rootPath`
- No Azure Blob involved — files stay on server filesystem

### `upload`
- User selects files/folder in the browser
- Files are uploaded directly to Azure Blob Storage
- Metadata and capsule records stored in Cosmos DB
- When terminal access is needed, the gateway **materializes** the workspace into a temp directory

## Upload Workspace Materialization for Terminal

When the user switches to Terminal with an upload workspace active:

1. Gateway enumerates workspace files from metadata store
2. Opens original content from Blob (via read URL)
3. Writes each file to a temp directory under `MESH_TERMINAL_UPLOAD_ROOT`
4. Caches a `.mesh-terminal-meta.json` marker for reuse
5. Node-pty shell opens in that materialized directory

This creates a bridge from Blob-backed records to a real shell-accessible directory tree.

## Workspace Identity

The canonical workspace identity flows through everything:

| Field | Where |
|-------|-------|
| `workspaceId` | Generated by browser on first open |
| `sessionId` | Per-upload session identifier |
| `folderName` | Human-readable folder slug |
| `rootPath` | Server-side path (local-path workspaces) |
| `sourceKind` | `"local-path"` or `"upload"` |

Because Gateway and Worker scale horizontally, `workspaceId` must be passed explicitly in every API call. Never rely on implicit worker in-memory state.

## Current Multiple-Truth Problem

The workspace identity is currently tracked in several places:

- `S.dirName` — browser-side folder name
- `S.tree` — browser-side file tree
- Local assistant workspace (gateway in-memory)
- Worker `workspaceState` (in-memory)
- Metadata store workspace summaries (Cosmos)
- Upload workspace IDs

Long-term goal: one canonical active workspace identity used by all views and APIs.

## Indexing Pipeline

### Phase 1: Initial Index

1. Browser discovers files in the selected folder
2. Browser tracks fingerprints (hash per file)
3. Browser computes diffs from previous state
4. Browser sends only changed/new files to `/api/assistant/workspace/sync`
5. Worker processes files: normalizes, filters, builds initial records
6. Event fires: `mesh-indexing-initial-ready`
7. Graph and chat become usable

### Phase 2: Background Enrichment

1. Worker continues processing remaining files
2. Upgrades initial records to full records (richer capsules)
3. Progress events: `mesh-indexing-background-progress`
4. Final event: `mesh-indexing-complete`
5. Graph refreshes

### Sync Request Shape

```javascript
{
  workspaceId: string,
  folderName: string,
  files: FileRecord[],       // only changed/new files
  deletedPaths: string[],
  append: boolean,
  mode: 'initial' | 'background' | 'single-file' | 'refresh',
  scanEpoch: number,
  complete: boolean,
}
```

This is a **diff-ingest model**, not full-resend. Only changes travel.

## Workspace State in Browser

`S.workspaceIndex` in `app-workspace.js`:

```javascript
{
  scanEpoch,
  knownFilesByPath,
  indexedFingerprintsByPath,
  pendingPaths,
  deletedPaths,
  initialIndexDone,
  backgroundIndexRunning,
  lastMode,
  // stats
  discovered,
  indexed,
  skipped,
  deleted,
}
```

Indexing is modeled as a phased, diff-based workflow — not a monolithic "upload everything" action.

## The `.mesh` File

When a folder is opened, the Worker generates a `.mesh` file at the workspace root via `provisionMeshFile()` and `buildMeshFileContent()` in `mesh-core/src/workspace-operations.js`.

This is a single intelligence file summarizing the workspace for quick context loading.
